# Efficient Streaming Language Models with Attention Sinks (StreamingLLM) (Xiao et al., 2023)

**arxiv**: 2309.17453 | **StreamingLLM 开山之作** | MIT / Meta | ICLR 2024

---

## 一、核心问题

大语言模型（LLM）在推理时需要缓存所有之前 token 的 KV 对（KV Cache），导致：
1. **内存线性增长**：序列越长，KV Cache 越大，最终 OOM
2. **无法处理超长输入**：即使模型理论上支持长上下文，实际推理时 KV Cache 也会溢出

朴素的解决方案是驱逐旧的 KV Cache 条目（滑动窗口），但作者发现这会导致**模型崩溃**——perplexity 暴增，生成质量急剧下降。

核心问题：**为什么简单地驱逐旧 KV 条目会摧毁模型？如何实现无限长度的流式推理？**

## 二、关键创新

### 2.1 Attention Sink 的发现

作者通过分析 LLM 的注意力分布，发现了一个普遍现象：**模型将大量注意力权重分配给序列的前几个 token**，无论这些 token 的语义内容如何。

具体表现：
- 在 Llama-2-70B 中，第一个 token 获得了约 60-90% 的注意力权重
- 前 4 个 token 是最主要的 attention sink
- 这种现象在所有层、所有 head 中都存在

**Attention Sink 的成因**：Softmax 函数要求所有注意力权重之和为 1。当模型不想将注意力分配给任何有意义的 token 时（即当前查询与所有已有 token 都不太相关），它需要一个"倾倒"多余注意力的地方。序列开头的 token（通常是 BOS）因为始终存在，成了天然的 sink。

### 2.2 为什么朴素驱逐会崩溃

当驱逐序列开头的 token 时：
- Attention sink 被移除
- 后续 token 被迫承接 sink 的角色，但它们的 KV 表示并非为此设计
- 注意力分布被严重扰乱，导致逐层误差累积
- 最终结果：perplexity 暴增数个数量级

### 2.3 StreamingLLM 方法

核心思想极其简单：**保留 attention sink token + 滑动窗口**。

$$\text{KV Cache} = \underbrace{\text{前 } s \text{ 个 sink token}}_{\text{始终保留}} + \underbrace{\text{最近 } w \text{ 个 token}}_{\text{滑动窗口}}$$

- $s$：sink token 数量（推荐 4）
- $w$：滑动窗口大小（如 4096）
- 总 KV Cache 大小固定为 $s + w$，不随序列长度增长

## 三、方法细节

### 3.1 算法流程

```
初始化：prefill 前 s+w 个 token 的 KV Cache

对于每个新生成的 token t：
  1. 计算 token t 对当前 KV Cache 的注意力
  2. 将 token t 的 KV 对加入 Cache
  3. 如果 Cache 大小 > s+w：
     驱逐第 s+1 个位置的 KV 对（最老的窗口内条目）
  4. 生成下一个 token
```

### 3.2 与相关方法的对比

| 方法 | KV Cache 大小 | 能否无限生成 | 实现复杂度 |
|------|-------------|------------|-----------|
| 全量 KV Cache | O(n) | 否（OOM） | 最低 |
| 朴素滑动窗口 | O(w) | 否（崩溃） | 最低 |
| **StreamingLLM** | O(s+w) | **是** | **最低** |
| 稀疏注意力 | O(n) 稀疏 | 是 | 高 |
| 重计算/分块 | O(w) + 重计算 | 是 | 中 |

### 3.3 Attention Sink 的实证分析

作者在多个模型和任务上验证了 attention sink 现象：
- **模型**：Llama-2 (7B/13B/70B), MPT, Falcon, Pythia
- **现象**：所有模型都存在 attention sink，且 sink 位置集中在序列开头
- **Sink token 的内容**：通常是 BOS token 或序列开头的前几个 token，内容无关紧要
- **层间差异**：浅层 sink 更强，深层稍弱但仍然显著

### 3.4 Rolling KV Cache 的实现

StreamingLLM 的 KV Cache 实现为环形缓冲区（ring buffer）：
- 固定大小的 $(s+w) \times d_{\text{kv}}$ 张量
- 新 token 写入最老窗口位置（循环覆盖）
- Sink token 的位置永远不被覆盖

## 四、实验结果

### 4.1 Perplexity 稳定性

在 PG-19 和 C4 上评估不同方法的 perplexity 随序列长度的变化：

| 方法 | 序列长度 4K | 8K | 16K | 32K |
|------|-----------|-----|------|------|
| 全量 KV Cache | 7.9 | 7.8 | 7.7 | 7.7 |
| 朴素滑动窗口 | 7.9 | **崩溃** | **崩溃** | **崩溃** |
| **StreamingLLM** | **7.9** | **7.8** | **7.7** | **7.7** |

StreamingLLM 在 400 万 token 的连续推理中维持稳定 perplexity，而朴素滑动窗口在几百 token 后就崩溃。

### 4.2 Sink 数量的影响

| Sink 数量 s | Perplexity |
|------------|-----------|
| 0 | 崩溃 |
| 1 | 稳定但偏高 |
| 2 | 较好 |
| **4** | **推荐值，接近全量** |
| 7 | 略好但边际收益递减 |

仅需 4 个 sink token 即可稳定推理。

### 4.3 下游任务

在长文本问答和摘要任务上，StreamingLLM 与全量 KV Cache 的质量差距很小，且能处理远超全量方法所能承载的序列长度。

## 五、影响

1. **无限长度推理成为可能**：StreamingLLM 使得 LLM 可以在不重启的情况下持续处理无限长的输入流，这对对话系统、流式日志分析等场景极为重要

2. **Attention Sink 的理论意义**：Sink 现象揭示了 Transformer 的一个架构缺陷——Softmax 的归一化约束迫使模型将注意力分配给不需要的 token。后续工作探索了替代 Softmax 的机制来缓解此问题

3. **KV Cache 压缩的新范式**：StreamingLLM 开创了"保留关键 token + 驱逐其余"的 KV Cache 管理范式，后续工作包括：
   - **H2O** (2023)：基于注意力分数动态选择驱逐哪些 KV 对
   - **Scissorhands** (2023)：基于注意力重要性剪枝 KV Cache
   - **Quest** (2024)：基于查询感知的 KV Cache 选择
   - **DMC** (2024)：动态内存压缩

4. **实践部署价值**：StreamingLLM 已被集成到多个推理框架（vLLM、TGI 等）中，是流式 LLM 推理的基础组件
