# From Governance Norms to Enforceable Controls: A Layered Translation Method for Runtime Guardrails in Agentic AI (Koch, 2026)

**arxiv**: 2604.05229 | **Agentic AI 治理转运行时防护** | 2026

---

## 一、核心动机

Agentic AI 系统（规划、工具调用、维护状态、多步执行轨迹）带来了与单轮生成式 AI 本质不同的治理问题：**关键风险在执行过程中涌现，而非仅在模型开发或部署时出现**。ISO/IEC 42001、ISO/IEC 23894、ISO/IEC 42005 等治理标准虽然高度相关，但它们本身不能直接产出可执行的运行时防护规则。本文提出一种分层翻译方法，将标准驱动的治理目标连接到四个控制层。

核心论点较为谨慎：**标准应指导控制放置的位置（架构、运行时策略、人工升级、审计），而运行时防护栏仅保留给那些可观察、确定性足够、时间敏感度高的控制**。

## 二、为什么不能直接翻译

### 类别不匹配

ISO 和 NIST 框架是关于管理、风险、生命周期、影响评估和治理的框架，本身不定义策略语言、动作模式或执行模型。把它们当作直接的防护栏规范，会混淆治理目标与技术机制。组织可以满足治理框架的要求，但部署的 Agent 仍然缺乏有意义的运行时控制。

### 运行时可执行性有限

部分治理规范可以较好地转化为技术控制（最小权限、范围授权、日志、审批门限、保留限制），但公平性、比例性、社会影响等要求需要上下文判断，无法安全地简化为确定性运行时规则。

### 防护栏本身不可靠

即使是非 Agentic 的 LLM 安全机制也是分层的、上下文敏感的和不完整的。对于 Agent，问题更难——决策跨工具调用和轨迹展开，而非单次输出。AgentSafetyBench 报告没有一个测试 Agent 的安全分数超过 60%。

### 控制层错位

范围过宽的 Agent 无法仅靠运行时过滤变得安全。预执行干预往往比后执行过滤更安全（因为某些危害一旦执行就不可逆）。有些控制应属于架构设计、模型选择、网络隔离、人工工作流设计。

## 三、治理到控制的翻译方法

### 四种不同的制品

1. **治理目标**（Governance Objective）：规范性目标，如问责、最小权限、影响感知、风险降低
2. **技术控制**（Technical Control）：实现目标的机制，如范围凭据、审批门限、日志、异常检测
3. **运行时防护栏**（Runtime Guardrail）：技术控制的子集，在执行时介入（允许/拒绝/延迟/升级/重塑动作）
4. **保障证据**（Assurance Evidence）：证明控制存在、是否执行、效果如何的制品

### 五步翻译方法

**Step 1**: 提取规范性目标——从标准或组织策略中提取

**Step 2**: 规范化为控制元组
$$\kappa = \langle a, x, r, \phi, \delta, \epsilon, o \rangle$$
其中 $a$ 是行为主体、$x$ 是动作类别、$r$ 是受保护资源、$\phi$ 是前置条件/上下文、$\delta$ 是控制决策（允许/拒绝/升级/仅记录/改写）、$\epsilon$ 是证据制品、$o$ 是问责所有者

**Step 3**: 评分运行时可执行性——使用运行时可执行性评分表

| 标准 | 高可执行性 | 低可执行性 |
|------|-----------|-----------|
| 伤害时机 | 执行前可阻止 | 事后才可评估 |
| 可观察性 | 所需状态可机器观察 | 关键上下文缺失 |
| 规则确定性 | 可写为确定性操作规则 | 需要开放性解释 |
| 判断负载 | 有限的社会/伦理判断 | 人类/上下文判断为核心 |
| 可逆性 | 难以撤销、干预紧急 | 可事后审计/纠正 |
| 证据清晰度 | 控制结果可清晰归因 | 证据模糊或归因困难 |

**Step 4**: 分配主要控制层——治理目标层、设计时层、运行时层、保障层

**Step 5**: 指定证据和所有权

## 四、案例研究：企业采购 Agent

| 需求 | 运行时可执行性 | 主要实现层 | 证据制品 |
|------|---------------|-----------|---------|
| 仅批准供应商可接收采购订单 | 高 | 设计时供应商目录范围 + 运行时白名单检查 | 签名决策追踪 |
| 超 5000 欧元需人工批准 | 高 | 运行时审批门限 | 批准令牌、不可变审批日志 |
| Agent 仅可访问采购必要系统 | 高 | 设计时最小权限凭据范围 + 运行时授权 | 已发行范围、访问日志 |
| 供应商排名应公平、可解释、可争议 | 低 | 设计时排名设计 + 保障审计 + 人工审核 | 定期审计报告 |
| 所有状态变更必须可归因和可重放 | 中高 | 运行时遥测 + 保障留存和重放管线 | 端到端追踪 |

关键洞察：前三项需求是好的运行时候选（相关状态在执行前可用且规则清晰），第四项不是（"公平可争议"太开放，无法安全编码为确定性预执行检查），第五项跨运行时和保障。

## 五、评估标准

五个维度：
1. **策略保真度**：翻译后的控制是否保留原始治理目标的含义？
2. **干预质量**：对有害动作的精确率和召回率
3. **轨迹覆盖**：控制能否推理部分路径、委托子 Agent、工具参数、累积状态？
4. **安全-效用权衡**：控制引入的延迟、任务完成损失、误升级负担
5. **证据完整性**：第三方能否确定哪个策略触发了、防护栏是否实际执行？

## 六、个人评价

这篇论文解决了一个非常实际的问题：组织有治理标准，但不知道如何将标准转化为可执行的技术控制。其核心贡献是：

1. **分层思维**：不是所有治理目标都适合运行时执行，强行这么做会要么过于保守（误杀良性动作）要么过于宽松（无法真正保护）
2. **控制元组**的规范化：使得控制变得具体、可检查、可审计
3. **运行时可执行性评分表**：提供了判断控制放置位置的实用工具

不足之处：
- 仍是方法+立场论文，缺乏实证验证
- 对 ISO 文档依赖公开描述而非逐条解读
- 评分表中的"高/低"判断仍是启发式而非严格标准
- 未涉及具体的运行时防护栏技术实现

这篇论文对 Agentic AI 的工程实践很有指导意义——它提醒我们，治理的落地需要分层、需要根据可执行性分配控制位置，而不是简单地"把标准编译成代码"。
