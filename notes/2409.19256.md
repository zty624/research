## 一、核心问题

RLHF（Reinforcement Learning from Human Feedback）是 LLM 对齐的关键技术，但其分布式执行面临根本挑战：

1. **RLHF 数据流复杂**：传统 RL 的数据流图中每个节点是简单 NN 计算，每条边是简单数据依赖。RLHF 中每个节点变成分布式 LLM 训练/生成程序（使用 3D 并行），每条边变成多对多组播（many-to-many multicast）——不同模型使用不同并行策略，数据需要 resharding
2. **单控制器范式低效**：传统 RL 框架（如 RLlib）用单控制器管理所有操作，对分布式 LLM 计算有巨大调度开销
3. **多控制器范式不灵活**：现有 RLHF 系统（DeepSpeed-Chat、OpenRLHF）用多控制器，每个设备独立运行，实现新算法需要修改所有相关模型代码，计算和数据传输耦合严重
4. **Actor 训练-生成转换开销大**：训练（计算密集，需大 MP）和生成（内存密集，需大 DP）使用不同并行策略，模型参数 resharding 代价高昂

核心洞察：**单控制器在节点间灵活协调数据传输，多控制器在节点内高效执行分布式计算**——两种范式应分层组合而非二选一。

## 二、方法

### 2.1 混合编程模型

**节点内（Intra-Node）：多控制器范式**
- 提供 `3DParallelWorker` 基类，封装分布式 LLM 计算（训练、推理、生成）
- 继承该基类的模型类（ActorWorker、CriticWorker 等）各自在分配的 GPU 上以多控制器方式执行
- 可无缝集成现有 LLM 框架（Megatron-LM、PyTorch FSDP、vLLM）
- 另提供 `FSDPWorker` 和 `ZeROWorker` 基类支持不同并行策略

**节点间（Inter-Node）：单控制器范式**
- 单个控制器进程协调所有模型的执行顺序和数据传输
- 提供 **Transfer Protocol** 机制：每个模型操作通过 `@register` 装饰器关联一个传输协议
- 传输协议包含 `collect` 函数（聚合输出数据）和 `distribute` 函数（分发输入数据）
- 提供 8 种预定义协议（3D_PROTO、DP_PROTO、ONE_TO_ALL 等），用户可扩展

**数据 Resharding 流程**：
1. 控制器调用源模型操作，获取 data futures
2. 控制器用源模型的 collect 函数聚合数据
3. 控制器用目标模型的 distribute 函数分发数据
4. 实际数据传输仅在 GPU 间进行，无中心瓶颈

**ResourcePool**：虚拟化 GPU 设备集合。同一 ResourcePool 实例的模型共置（colocate）在同一 GPU 集上，不同实例放在不同设备。

**异步执行**：不同设备集上的模型在输入就绪后自动并行执行；共置模型按调用顺序顺序执行。

### 2.2 灵活的 RLHF 算法实现

由于计算与数据传输解耦，实现新算法只需几行代码：

```python
# PPO: 8 行
for prompts in dataloader:
    batch = actor.generate_sequences(prompts)
    batch = critic.compute_values(batch)
    batch = reference.compute_log_prob(batch)
    batch = reward.compute_reward(batch)
    batch = compute_advantages(batch, "PPO")
    critic.update_critic(batch, loss_func="PPO")
    actor.update_actor(batch, loss_func="PPO")

# Safe-RLHF: PPO + 5 行（添加 cost model 和 pretrain loss）
# ReMax: PPO 修改（额外生成 + 移除 critic）
```

对比多控制器实现，后者需要在每个模型的代码中显式管理 send/recv 和 all_gather 操作，修改一个节点需改变所有依赖节点的实现。

### 2.3 3D-HybridEngine：Actor 训练-生成高效转换

Actor 模型在训练和生成阶段使用不同 3D 并行策略：
- 训练：$(p, t, d)$——需要大 MP（大 $t$ 和 $p$）充分利用计算
- 生成：$(p_g, t_g, d_g, d)$——需要大 DP（大 $d_g$）充分利用内存带宽

约束：$N_a = p \times t \times d = p_g \times t_g \times d_g \times d$，因此 $d_g = \frac{pt}{p_g t_g}$。

**零冗余 Resharding 关键设计**：

朴素方法（HybridFlow-V）：训练和生成使用相同的并行分组方法（连续 rank 分配 TP 和 PP），仅改变并行大小。这导致某些 GPU 上训练和生成的权重无重叠，需要额外内存保存训练权重用于后续迭代。

优化方法（HybridFlow）：重新设计生成阶段的并行分组方式：
- **生成 TP/PP 组**：按 $t/t_g$ 和 $p/p_g$ 的间隔选取 rank，而非连续分配
- **Micro DP 组**：沿生成 TP 或 PP 维度顺序分配 rank

这种分组使得**每个 GPU 上训练和生成的权重有重叠**，可在生成时复用训练权重，消除内存冗余。

**通信量对比**（$M$ 为模型大小，$N_a$ 个 GPU）：

| 方案 | 通信量 | 峰值内存 | 冗余 |
|------|--------|---------|------|
| DeepSpeed-Chat | $\frac{tpd - 1}{N_a} M$ | $M$ | $\frac{1}{tpd} M$ |
| HybridFlow-V | $\frac{tp - 1}{tp} M$ | $M$ | $\frac{1}{tp} M$ |
| **HybridFlow** | $\frac{tp - t_g p_g}{t_g p_g tp} M$ | $\frac{1}{t_g p_g} M$ | **0** |

HybridFlow 通信量最小且内存零冗余。

**3D-HybridEngine 工作流**：
1. 在 micro DP 组内 all-gather 模型参数（生成阶段）
2. 加载 prompt 并自回归生成
3. 在 micro DP 组内 all-gather 生成结果
4. 按训练并行配置重新切分模型参数
5. 执行 actor 训练

### 2.4 自动设备映射算法

给定 RLHF 数据流图 $D$、LLM 列表 $L$、总 GPU 数 $N$，算法搜索最优设备放置和并行策略：

1. 枚举所有可能的模型放置方案 $P$（Bell 分区问题，PPO 4 个模型有 15 种放置）
2. 对每种放置，计算最小 GPU 分配 $A_{\min}$（基于内存约束）
3. 从 $A_{\min}$ 开始枚举所有可行 GPU 分配
4. 对每种分配，使用 `auto_parallel` 模块搜索最优并行策略
5. 用 `d_cost` 模块估计端到端延迟，选择最优方案

延迟计算规则：
- 同一共置集内的模型在同一阶段顺序执行（延迟相加）
- 不同共置集的模型可并行执行（延迟取最大值）

## 三、实验结果

### 3.1 实验设置

- 硬件：16 台机器 × 8 NVIDIA A100-80GB GPU（128 GPU 总计）
- 模型：Llama 7B/13B/34B/70B
- 算法：PPO、ReMax、Safe-RLHF
- 数据集：Dahoas/ful-hh-rlhf，prompt 和 response 长度均为 1024

### 3.2 PPO 吞吐量

| Actor 大小 | 对比 DeepSpeed-Chat | 对比 OpenRLHF | 对比 NeMo-Aligner |
|-----------|-------------------|-------------|------------------|
| 7B | 1.68×~8.63× | 1.53×~2.56× | — |
| 13B | 2.70×~18.96× | 2.49×~3.66× | — |
| 34B | 2.41×~20.57× | 2.14×~4.80× | — |
| 70B | 5.17×~17.98× | 6.46×~9.78× | — |

HybridFlow 在所有模型大小和 GPU 规模上均显著优于所有基线。

### 3.3 ReMax 吞吐量

| Actor 大小 | 对比 DeepSpeed-Chat | 对比 OpenRLHF | 对比 NeMo-Aligner |
|-----------|-------------------|-------------|------------------|
| 7B | 1.71×~12.87× | — | — |
| 13B | 2.49×~18.47× | — | — |
| 34B | 2.20×~19.76× | — | — |
| 70B | 4.89×~16.86× | — | — |

### 3.4 3D-HybridEngine 消融

- 零冗余 resharding 对大模型（70B）收益显著：避免 140GB 权重传输，节省最高 36.4% 迭代时间
- 不同并行配置的训练/生成吞吐量对比证实：训练用大 MP、生成用大 DP 是最优策略
- Mamba 骨干式的时间卷积对 TTT-Linear 帮助更大（因其隐藏状态表达力弱于 MLP）

### 3.5 自动映射 vs 手动映射

自动映射算法找到的配置接近手动调优的最优配置，在 GPU 数量少时甚至超过手动配置。

## 四、个人评价

### 影响

1. **混合编程模型**：首次系统地将单控制器和多控制器范式分层组合用于 RLHF，解决了灵活性与效率的根本矛盾。这一设计理念可推广到其他复杂分布式 ML 工作流
2. **3D-HybridEngine**：零冗余的 actor resharding 方案是关键工程贡献。通过巧妙地重新排列并行分组方式，使训练和生成阶段的权重在每个 GPU 上有重叠，消除了内存冗余并显著降低通信量
3. **Transfer Protocol 抽象**：将数据 resharding 从模型计算中解耦，用 collect/distribute 函数对抽象数据传输。这使得添加新模型或修改并行策略不影响其他模型的代码
4. **自动设备映射**：为 RLHF 提供了自动化的资源分配和并行策略搜索，减少了人工调优的负担
5. **开源影响**：HybridFlow 是 verl 框架的核心，在开源 RLHF 社区有广泛影响

### 局限

1. **假设同构 GPU**：自动映射算法假设所有 GPU 同构，不支持异构集群
2. **Co-location 的 OOM 风险**：共置模型顺序执行避免了并发 OOM，但内存利用率不高——一个模型运行时其他模型的 GPU 内存空闲
3. **仅验证 PPO 类算法**：虽然展示了 ReMax 和 Safe-RLHF，但未验证 DPO、GRPO 等不需要 4 个模型的 RLHF 变体
4. **Transfer Protocol 有限的自动化**：8 种预定义协议覆盖了常见场景，但新的并行策略组合可能需要手动实现新协议
5. **vLLM 版本较旧**：使用 vLLM 0.3.1，后续版本的优化（如连续批处理）未利用
6. **通信开销分析不够深入**：虽然量化了 resharding 通信量，但未分析数据传输与计算的重叠（overlap）策略
7. **冷启动问题**：首次迭代需要从检查点加载模型，预热时间未在评估中体现
8. **缺乏与更新系统的对比**：如 OpenRLHF 后续版本和其他 RLHF 框架的改进
