# Transformer-XL: Attentive Language Models Beyond a Fixed-Length Context (Dai et al., 2019)

**arxiv**: 1901.02860 | **长上下文 Transformer 开山之作** | CMU & Google Brain | ACL 2019

---

## 一、核心问题

Vanilla Transformer 做语言建模时，必须将长序列切分成固定长度的 segment 逐段训练。这带来两个根本性缺陷：

1. **上下文窗口固定**：每个 segment 完全独立处理，模型无法利用前一段的信息来理解当前段。例如，当关键信息出现在第 1 段而需要预测第 3 段的 token 时，模型彻底"失忆"
2. **位置编码的段级断裂**：绝对位置编码在每个段内从 0 重新开始，模型无法区分"当前段的位置 3"和"上一段的位置 3"，导致跨段推理不可能

后果：vanilla Transformer 能建模的最长依赖 = segment 长度，RNN 虽然有梯度消失问题但理论上信息可以无限传递。

## 二、方法

### 2.1 Segment-Level Recurrence（段级循环）

核心思想：**不丢弃上一段的隐藏状态，而是缓存起来作为下一段的扩展上下文**。

设第 $\tau$ 段的隐藏状态序列为 $s_\tau = [h_{\tau,1}, \ldots, h_{\tau,L}]$，第 $\tau+1$ 段的计算：

$$\tilde{h}_{\tau+1} = [\text{sg}(s_\tau), h_{\tau+1}]$$

其中 $\text{sg}(\cdot)$ 表示 stop-gradient（不反传梯度到上一段），$[\cdot, \cdot]$ 沿序列维度拼接。注意力的 key 和 value 来自拼接后的 $\tilde{h}_{\tau+1}$，query 仅来自当前段 $h_{\tau+1}$。

**关键细节**：
- 缓存的隐藏状态 detach 梯度，避免跨段的梯度爆炸/内存爆炸
- 每层都独立缓存：第 $l$ 层的 memory 来自上一 segment 第 $l$ 层的输出
- 有效上下文长度 = segment 长度 $\times$ 层数（信息通过层间传递可以跨越更远的距离）

### 2.2 Relative Positional Encoding（相对位置编码）

段级循环引入新问题：缓存了上一段的状态后，key 来自两段，但绝对位置编码无法表达跨段的相对距离。

解决方案：**把位置信息从 embedding 层移到 attention 计算中，用相对位置代替绝对位置**。

原始 attention 项（绝对位置）：
$$A_{ij} = (E_{x_i} + U_i)^\top (E_{x_j} + V_j)$$

Transformer-XL 将其改写为四项之和：
$$A_{ij} = \underbrace{E_{x_i}^\top W_q^{} W_{k,E} E_{x_j}}_{\text{(a) content-content}} + \underbrace{E_{x_i}^\top W_q^{} W_{k,R} R_{i-j}}_{\text{(b) content-position}} + \underbrace{u^\top W_{k,E} E_{x_j}}_{\text{(c) bias-content}} + \underbrace{v^\top W_{k,R} R_{i-j}}_{\text{(d) bias-position}}$$

- **(a)** query 内容与 key 内容的交互
- **(b)** query 内容与 key 相对位置的交互——$R_{i-j}$ 是可学习的相对位置 embedding
- **(c)** query 的全局偏置与 key 内容的交互——$u$ 是可学习参数，替代 $U_i$
- **(d)** query 的全局偏置与 key 相对位置的交互——$v$ 是可学习参数，替代 $V_j$

**为什么这样改就解决了跨段问题？** 因为 $R_{i-j}$ 只依赖相对偏移量 $i-j$，与绝对位置无关。无论 key 来自当前段还是缓存段，只要相对距离相同，位置编码就相同——这保证了跨段的泛化性。

### 2.3 高效实现

直接计算 (b) 项需要对每个 query 计算 $E_{x_i}^\top W_q W_{k,R} R_{i-j}$，复杂度 $O(L \times (L+M) \times d)$。Transformer-XL 的技巧：

1. 先计算 $W_q^\top E_{x_i}$ 得到 query 向量 $q_i$
2. 将 $W_{k,R} R_{i-j}$ 按位置平移排列成矩阵
3. 用矩阵乘法批量计算

这样相对位置项的计算可以复用标准 attention 的框架，实际开销很小。

## 三、实验结果

### 3.1 语言建模

在 WikiText-103 上的 perplexity：

| 模型 | PPL |
|------|-----|
| LSTM + Mositch | 48.7 |
| LSTM + heuristic softmax | 46.5 |
| Vanilla Transformer (短上下文) | 54.5 |
| Transformer-XL | **44.4** |

Transformer-XL 以显著优势超越所有 RNN 和 vanilla Transformer 基线。

### 3.2 长距离依赖建模能力

论文设计了专门的评估：让模型处理需要长距离依赖的任务（如延迟 copy）。

- Transformer-XL 学到的依赖长度比 RNN 长 **80%**
- 比 vanilla Transformer 长 **1800%**（18 倍）

### 3.3 消融实验

| 配置 | PPL |
|------|-----|
| 完整 Transformer-XL | 44.4 |
| 去掉 recurrence（仅相对位置编码） | 46.6 |
| 去掉相对位置编码（仅 recurrence + 绝对编码） | 46.9 |
| 两者都去掉（= vanilla Transformer） | 48.3 |

两个组件都重要，recurrence 贡献略大于相对位置编码，两者互补。

### 3.4 推理速度

由于缓存机制，推理时不需要逐 token 重新计算整个上下文，速度比 vanilla Transformer 快 **1800 倍**（序列越长优势越大）。

## 四、我们的实验（50_transformer_xl.py）

实验在 Copy Task 上对比三种配置：
- **Vanilla Transformer**：固定上下文窗口，绝对位置编码
- **Transformer-XL（有 memory）**：segment-level recurrence + 相对位置编码
- **Transformer-XL（无 memory）**：仅相对位置编码，不缓存前段隐藏状态

关键设置：
- 序列长度 80，延迟 10（需记住前 10 个 token 并在末尾复现）
- segment 长度 40，memory 长度 40
- 模型：d_model=64, n_heads=2, n_layers=2

实验还测试了不同序列长度（60/100/140）下的扩展性，观察 Transformer-XL 是否在更长序列上保持优势。

可视化包括：训练曲线对比、序列长度 scaling 曲线、绝对 vs 相对位置编码的注意力模式对比。

## 五、个人评价

Transformer-XL 是长上下文建模的关键里程碑：

1. **Segment-level recurrence 简洁优雅**：不修改架构，只是缓存和复用隐藏状态，却打破了固定上下文窗口的硬限制。这种"用 memory 扩展上下文"的思想后来被广泛继承
2. **相对位置编码是深层需求**：不是为了"更好"的位置表示，而是因为段级循环与绝对位置编码在逻辑上不兼容。这个分析很深刻
3. **深远影响**：
   - **XLNet**（2019）直接基于 Transformer-XL 的 recurrence 机制做预训练
   - **Memorizing Transformers**（2022）用 kNN 检索扩展 memory，是 recurrence 的自然延伸
   - 现代 LLM 的 KV cache 可以看作 recurrence 机制的一种实现（虽然不跨 segment 边界）
4. **局限性**：
   - Recurrence 是有界的——只能缓存固定长度的 memory，不是真正的无限上下文
   - 缓存增加了内存开销
   - 后续工作（如 Compressive Transformer、Memorizing Transformer）尝试压缩或检索 memory 来进一步扩展

Transformer-XL 的核心贡献是证明了：**Transformer 不必受限于固定长度上下文，通过巧妙的缓存机制可以突破这一限制**。这一思路至今仍是长上下文研究的重要方向。
