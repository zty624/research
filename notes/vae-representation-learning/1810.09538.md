# Pyro: Deep Universal Probabilistic Programming (Bingham et al., 2019)

**arxiv**: 1810.09538 | **概率编程语言** | Uber AI Labs | JMLR 2019

---

## 一、核心动机

概率编程语言（PPL）旨在降低概率模型的开发门槛，但现有系统存在根本性矛盾：

1. **表达力 vs. 可扩展性**：通用 PPL（如 Church、Venture）能表达任意可计算概率分布，但推理算法无法扩展到大数据和高维模型
2. **可扩展性 vs. 灵活性**：可扩展的系统（如 Stan）限制模型类别，或无法支持模型特定的推理算法
3. **灵活性 vs. 极简性**：灵活的系统语法复杂、认知负担高

Pyro 的目标是同时实现四个设计原则——**表达力、可扩展性、灵活性、极简性**。

## 二、核心设计

### 2.1 极简语言原语

Pyro 嵌入在 Python 中，仅增加两个原语：

- **`pyro.sample(name, fn, obs=...)`**：标注随机变量。`obs` 关键字参数指定观测值，用于定义条件分布
- **`pyro.param(name, init=...)`**：注册可学习参数，由推理算法更新

Pyro 模型可以包含任意 Python 控制流（循环、条件、递归），随机变量的存在可依赖于其他变量的值——这比静态计算图框架灵活得多。

### 2.2 随机变分推理（SVI）作为核心推理算法

SVI 是 Pyro 的主要推理方法：

- 用随机梯度下降优化近似后验 $q_\phi(z|x)$（称为 guide）与真实后验 $p(z|x)$ 之间的散度
- 利用 PyTorch 的 GPU 加速张量运算和反向模式自动微分
- 通过 mini-batch 随机梯度估计实现大数据扩展

Guide 的设计：
- Guide 必须与 model 有相同的输入参数
- Guide 中每个无观测的 `sample` 语句必须有对应的 `sample` 语句
- 用户可自由设计近似后验的条件独立结构
- Guide 不能依赖 model 内部的值（与 webPPL/Anglican 不同）

### 2.3 Poutine：可组合的效果处理器

Pyro 的核心抽象是 **Poutine**——一组可组合的效果处理器（effect handlers），用于检查和修改概率程序的行为：

- **`trace`**：记录执行轨迹（输入、输出、函数调用）
- **`condition`**：将未观测的 sample 站点设为指定值
- **`replay`**：从一个 trace 中回放随机选择到另一个执行中
- **`block`**：阻止特定站点的效果处理

Poutine 的关键优势：**将推理算法的实现与语言细节分离**。推理算法通过组合效果处理器来操作概率程序，而无需修改程序本身。这使得添加新推理算法变得容易。

### 2.4 其他推理算法

- **NUTS**（No-U-Turn Sampler）：HMC 的自适应变体
- **重要性采样**
- **HMC**（Hamiltonian Monte Carlo）
- 支持自定义推理算法

### 2.5 与其他 PPL 的对比

| 系统 | 表达力 | 可扩展性 | 灵活性 | 极简性 |
|------|--------|---------|--------|--------|
| Stan | 静态控制流 | CPU | 自动化 | DSL |
| Church | 动态 | 无 | 自动化 | Scheme |
| Venture | 动态 | 无 | 灵活 | 自定义 VM |
| webPPL | 动态 | CPU | 部分 | JavaScript |
| Edward | 静态 | CPU/GPU | 部分 | TensorFlow |
| **Pyro** | **动态** | **CPU/GPU** | **灵活** | **Python/PyTorch** |

Pyro 是当时唯一同时实现动态控制流 + GPU 加速 + 灵活推理 + Python 生态的 PPL。

## 三、实验结果

### VAE

Pyro 实现的 VAE 与纯 PyTorch 实现性能对比：

| 隐变量维度 | 隐层维度 | PyTorch (ms) | Pyro (ms) |
|-----------|---------|-------------|-----------|
| 10 | 400 | 3.82 | 6.79 |
| 30 | 400 | 3.73 | 6.67 |
| 10 | 2000 | 7.65 | 10.14 |
| 30 | 2000 | 7.66 | 10.19 |

- 每次 update 耗时约 1.5-1.8 倍（由于 trace 开销），可接受
- 收敛到相同 test ELBO
- 功能更完整：自动处理 trace、条件化等

### Deep Markov Model (DMM)

- 实现 DMM + IAF（Inverse Autoregressive Flow）guide
- 在 IAF 数量增加时，test ELBO 持续改善（-6.93 → -6.80）
- 仅几行代码即可将 DMM 的近似后验替换为更强大的 IAF
- 验证了 Pyro 处理复杂层次化模型的能力

### 对 PyTorch 生态的贡献

Pyro 团队向 PyTorch 贡献了 **PyTorch Distributions** 库——这是 PyTorch 核心库的重要组成部分，受 TensorFlow Distributions 启发。

## 四、个人评价

Pyro 是概率编程语言领域的重要工作，在实用性和学术影响力之间取得了好的平衡：

1. **Poutine 抽象是核心创新**：效果处理器将模型定义、推理算法和运行时行为解耦，使得推理算法可以像"插件"一样组合。这一设计后来被 NumPyro 直接继承，成为 PPL 设计的经典范式
2. **Python + PyTorch 生态优势**：利用 PyTorch 的自动微分和 GPU 加速，降低了入门门槛。同时 Pyro 团队反向贡献了 PyTorch Distributions 库
3. **SVI 优先的策略是务实选择**：MCMC 虽然精确但不可扩展，SVI+深度网络在大数据场景下更实用。Pyro 两者都支持但侧重 SVI
4. **Guide 机制的双刃剑**：Guide 给用户设计近似后验的自由度，但也要求用户理解变分推理——这比 Stan 的"全自动推理"门槛更高

**局限性**：
- 运行时 trace 开销使每次迭代比纯 PyTorch 慢约 50-80%
- 动态图带来灵活性，但也限制了编译优化空间（后续 NumPyro 用 JAX JIT 弥补）
- 某些高级推理算法（如 SMC、粒子 MCMC）的支持不如专业 PPL 完善
- Guide 设计需要用户有较强的概率建模背景，容易因 guide 与 model 不匹配导致推理质量差

Pyro 在工业界被广泛采用（Uber 内部、生物医药等），也是学术研究中构建复杂概率模型的常用工具。它和 NumPyro 一起构成了 Python 生态中最重要的概率编程基础设施。
