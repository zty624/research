# Towards Using Behavior Trees in Industrial Automation Controllers (Sidorenko et al., 2024)

**arxiv**: 2404.14030 | **关键词**: 行为树, PLC, 工业自动化, 技能编程, IEC 61131, IEC 61499 | DFKI, University of Kaiserslautern-Landau | 2024

## 一、核心问题

工业 4.0 下的信息物理生产系统（CPPS）要求控制软件具备**模块化、灵活性和快速重配置**能力，但当前 PLC（可编程逻辑控制器）软件存在以下问题：

1. **缺乏模块化**：PLC 程序通常高度集成，难以复用和重配置
2. **关注点未分离**：硬件相关功能与应用逻辑紧密耦合，修改一处需改动整个程序
3. **AI 与低层控制的鸿沟**：AI 驱动的任务导向框架与实时 PLC 控制之间存在集成缺口
4. **非程序员无法参与**：应用工程师（领域专家）需要编程技能才能修改控制逻辑

本文提出将**行为树（Behavior Trees, BTs）**集成到 PLC 程序中，分离硬件功能与应用逻辑，提升模块化和可重配置性。

## 二、方法

### 2.1 行为树模型

行为树是编码智能体行为的控制结构，将环境状态映射到动作。核心概念：

- **控制流节点**：Sequence（$\rightarrow$）、Fallback（?）
- **执行节点**：Action（$\Rightarrow$）、Condition（$\#$）
- 每个节点返回三种状态之一：**Success / Failure / Running**

| 节点类型 | 成功条件 | 失败条件 | 运行条件 |
|----------|----------|----------|----------|
| Sequence | 所有子节点成功 | 任一子节点失败 | 任一子节点运行中 |
| Fallback | 任一子节点成功 | 所有子节点失败 | 任一子节点运行中 |
| Action | 完成时 | 无法完成时 | 执行中 |
| Condition | 条件为真 | 条件为假 | 永不 |

BT 的递归结构使得每个子树既是子节点的树，又是父节点的动作节点。优先级从左到右递减，抽象级别从上到下递增。

### 2.2 PLCopen Common Behavior Model

PLCopen 定义了标准化的功能块（FB）行为模型，核心是 **ETrigA**（Edge Triggered Action）：
- 输入：`xExecute`（启动）、`xAbort`（中止）
- 输出：`xDone`（完成）、`xError`（错误）、`xBusy`（执行中）

关键观察：**BT 执行语义与 PLCopen ETrigA FB 高度一致**——两者都用 3 个反馈信号表示执行状态（Running/Success/Failure vs Executing/Done/Error），这为集成提供了自然基础。

### 2.3 三种集成策略

**策略 1：外部 BT 框架 + 适配器**
- BT 在外部平台（如 ROS2 + BehaviorTree.CPP）运行
- PLC 端 PLCopen FB 通过 ctrlX Data Layer 暴露接口
- ETrigA 适配器（ROS 节点）在 PLC 和 BT 之间同步
- 优势：完全解耦 PLC 和 BT 侧；劣势：通信开销

**策略 2：IEC 61131 内嵌 BT 库**
- 将 BT 节点实现为继承自 ETrigA 的功能块
- BT tick 信号映射为 FB 的 `xExecute`/`xAbort` 输入
- BT 状态映射为 FB 的 `xDone`/`xError`/`xBusy` 输出
- Sequence 节点采用**非阻塞循环实现**（Algorithm 1），遍历子节点状态直到第一个 Running 节点
- 适配 IEC 61131 的**同步循环执行模型**，避免递归和阻塞

**策略 3：IEC 61499 事件驱动 BT**
- 利用 IEC 61499 的事件驱动执行模型
- 每个 BT 节点建模为异步进程，通过事件通信
- BT 控制流节点使用 **ECC（Execution Control Chart）**实现状态机
- 事件名称：`PARENT.x`（与父节点通信）、`CHILD_x.x`（控制子节点）

### 2.4 BT 驱动的 PLC 软件设计

核心思想：**分离关注点**——
- **底层**：专业开发者创建硬件相关的 PLCopen FB 库
- **上层**：应用工程师使用低/无代码 BT 平台组合 FB，构建应用逻辑

BT 的优势：
1. 设计直观，非程序员可理解
2. 模块化，修改一个子树不影响其他部分
3. 支持 top-down（从高层规格生成 BT）和 bottom-up（从组件组合 BT）两种开发方式
4. BT 应用逻辑可由 AI 规划方法自动生成

## 三、实验结果

### 应用示例：机器人 + 线性轴
- 机器人由 ROS2 控制，线性轴由 PLCopen 运动控制 FB + IEC 61131 PLC 控制
- BT 控制线性轴的使能、复位和移动，条件节点从 `MC_ReadStatus` FB 获取状态
- 应用逻辑完全编码在 BT 中，修改 BT 不需要触碰底层 FB

### IEC 61131 BT 库验证
- 成功在 ctrlX CORE 平台实现 BT 功能块库
- 非阻塞 Sequence 节点在 PLC 循环中正常工作
- BT FB 的调用顺序对确定性行为重要

### IEC 61499 事件驱动 BT 验证
- 在 4DIAC IDE 中实现了事件驱动的 BT Sequence FB
- ECC 状态机正确实现了 BT 的控制流语义

## 四、个人评价

### 影响
- **工业自动化的新范式**：将游戏 AI 和机器人领域流行的行为树引入 PLC 编程，可能改变工业控制软件的开发方式
- **关注点分离**：底层 FB 与上层 BT 逻辑的分离，使应用工程师无需深入 PLC 编程即可重配置系统
- **AI 集成桥梁**：BT 作为 AI 规划输出与 PLC 执行之间的中间表示，为工业 AI 部署提供路径
- **跨标准兼容**：同时支持 IEC 61131 和 IEC 61499 两种 PLC 标准

### 局限
- **仅概念验证**：应用示例较简单（单轴运动控制），未在复杂工业场景中大规模验证
- **性能开销**：外部 BT 框架的通信延迟和内嵌 BT 库的循环扫描开销未定量评估
- **大型 BT 的扩展性**：非阻塞 Sequence 需遍历子节点，对非常大的 BT 可能需要分拆
- **IEC 61499 采用率低**：工业界仍以 IEC 61131 为主，策略 3 的实用价值受限
- **缺乏定量对比**：未与传统 PLC 设计方法进行开发时间、可维护性等定量对比
