# Toolformer: Language Models Can Teach Themselves to Use Tools (Schick et al., 2023)

**arxiv**: 2302.04761 | **Toolformer 开山** | Meta AI | arXiv 2023

---

## 一、核心问题

大型语言模型虽然展现出强大的 few-shot 和 zero-shot 能力，但存在几个根本性缺陷：(1) 无法获取最新信息，容易产生事实幻觉；(2) 精确数学计算能力差；(3) 不理解低资源语言；(4) 缺乏时间感知。这些缺陷无法单纯通过扩大模型规模解决。

已有方法要么依赖大量人类标注（如 LaMDA），要么仅在任务特定设置中使用工具（如给定特定提示时才调用），**缺乏一种让 LM 自主决定何时、如何使用哪个工具的方法**。

核心问题：**能否让语言模型以自监督方式学会使用外部工具，且不损失其通用语言建模能力？**

## 二、方法

Toolformer 的核心思路：让 LM 自己决定哪些 API 调用对预测未来 token 有帮助，然后在有用的 API 调用上微调自己。

### API 调用表示

每个 API 调用表示为元组 $c = (a_c, i_c)$，其中 $a_c$ 为 API 名称，$i_c$ 为输入。线性化表示：

- 不含结果：$e(c) = \texttt{<API>} \; a_c(i_c) \; \texttt{</API>}$
- 含结果：$e(c, r) = \texttt{<API>} \; a_c(i_c) \to r \; \texttt{</API>}$

实践中使用 `[`、`]`、`->` 替代特殊 token，**无需修改 LM 词表**。

### 三步自监督流程

**Step 1: 采样 API 调用**
- 为每个 API 编写 few-shot 提示 $P(x)$，教 LM 在文本中插入 API 调用
- 对文本 $x$ 的每个位置 $i$，计算 $p_i = p_M(\texttt{<API>} \mid P(x), x_{1:i-1})$
- 保留概率超过阈值 $\tau_s$ 的位置（最多 $k$ 个），每个位置采样 $m$ 个 API 调用
- 若模型未生成 `</API>` token 则丢弃该样本

**Step 2: 执行并过滤 API 调用**
- 执行所有 API 调用获取结果 $r_i$
- 使用**自监督过滤**：计算加权交叉熵损失

$$L_i(z) = -\sum_{j=i}^{n} w_{j-i} \cdot \log p_M(x_j \mid z, x_{1:j-1})$$

权重衰减函数：$\tilde{w}_t = \max(0, 1 - 0.2 \cdot t)$，归一化后使用。确保 API 调用提供的信息在其附近最有帮助。

比较两种损失：
- $L_i^+ = L_i(e(c_i, r_i))$：有 API 调用及结果时的损失
- $L_i^- = \min(L_i(\varepsilon), L_i(e(c_i, \varepsilon)))$：$\min($无 API 调用损失, 有调用无结果损失$)$

**注意**：过滤时将 API 调用及结果作为 prefix 而非插入中间位置，因为模型尚未微调，插入中间会打断文本流、增加 perplexity。

保留条件：$L_i^- - L_i^+ \geq \tau_f$，即 API 调用使未来 token 的预测显著变好。

不同过滤阈值 $\tau_f$ 下保留的样本数量差异巨大（以 QA 工具为例：$\tau_f=0.5$ 保留 51,987 个，$\tau_f=2.0$ 仅保留 5,135 个）。

**Step 3: 微调**
- 将过滤后的 API 调用插入原文，形成增强数据集 $C^*$
- 在 $C^*$ 上用标准语言建模目标微调 M
- 微调超参：batch size 128，学习率 $1 \times 10^{-5}$，线性 warmup 前 10%

关键设计：$C^*$ 与原始 $C$ 包含完全相同的文本内容（仅插入了 API 调用），因此微调不损失通用能力。

### 五种工具

| 工具 | 描述 | 底层模型 | 输入约束 |
|------|------|---------|---------|
| Question Answering | 回答事实性问题 | Atlas (检索增强 LM，微调于 Natural Questions) | 事实性问题 |
| Calculator | 四则运算，保留两位小数 | 简单计算器 | 数值表达式（至少含 3 个数字的文本） |
| Wikipedia Search | BM25 检索 Wikipedia 摘要 | BM25 + KILT Wikipedia | 搜索词 |
| Machine Translation | 任意语言→英语翻译 | NLLB-600M (200 种语言，fastText 自动检测源语言) | 非英文文本 |
| Calendar | 返回当前日期 | 系统日历 | 无输入 |

**启发式预筛选**：为减少计算成本，对部分 API 使用启发式规则预筛选 CCNet 子集（如 Calculator 只处理含至少 3 个数字的文本）。

### 推理

解码时，模型自主决定何时调用 API。当模型生成 `->` token 时暂停解码，调用对应 API 获取结果，插入结果后继续解码。使用 $k=10$ 的 top-k 解码以增加工具调用倾向——即 `<API>` token 只要是前 $k$ 个最可能的 token 之一就会触发调用。每个输入最多一次 API 调用以避免死循环。

## 三、实验结果

### LAMA（知识探测，使用 QA 工具）

| 模型 | SQuAD | Google-RE | T-REx |
|------|-------|-----------|-------|
| GPT-J (6.7B) | 17.8 | 4.9 | 31.9 |
| GPT-J + CC | 19.2 | 5.6 | 33.2 |
| Toolformer (disabled) | 22.1 | 6.3 | 34.9 |
| **Toolformer** | **33.8** | **11.5** | **53.5** |
| OPT (66B) | 21.6 | 2.9 | 30.1 |
| GPT-3 (175B) | 26.8 | 7.0 | 39.8 |

Toolformer (6.7B) 使用 QA 工具后**超越 GPT-3 (175B)**，尽管参数量仅为后者的 1/26。98.1% 的例子中模型自主选择使用 QA 工具，0.7% 使用其他工具，1.2% 不使用任何工具。

### 数学推理（使用 Calculator 工具）

| 模型 | ASDiv | SVAMP | MAWPS |
|------|-------|-------|-------|
| GPT-J | 7.5 | 5.2 | 9.9 |
| GPT-J + CC | 9.6 | 5.0 | 9.3 |
| Toolformer (disabled) | 14.8 | 6.3 | 15.0 |
| **Toolformer** | **40.4** | **29.4** | **44.0** |
| OPT (66B) | 6.0 | 4.9 | 7.9 |
| GPT-3 (175B) | 14.0 | 10.0 | 19.8 |

97.9% 的例子中模型自主选择使用计算器工具。Toolformer (disabled) 也比 GPT-J 提升明显，因为微调数据中包含大量 API 调用及结果，间接提升了模型的数学能力。

### 问答（Wikipedia Search 工具）

| 模型 | WebQS | NQ | TriviaQA |
|------|-------|----|----------|
| GPT-J | 18.5 | 12.8 | 43.9 |
| **Toolformer** | **26.3** | **17.7** | **48.8** |
| GPT-3 (175B) | 29.0 | 22.6 | 65.9 |

99.3% 的例子使用 Wikipedia Search。仍落后 GPT-3 (175B)，主要因为 BM25 搜索引擎过于简单，且模型无法交互式调整查询。

### 多语言问答（使用 MT 工具）

MLQA 基准（英文段落 + 各语言问题）：MT 工具使用率从 63.8% 到 94.9% 不等（印地语仅 7.3%）。但由于在 CCNet 上微调导致分布偏移，Toolformer 并未一致超越 vanilla GPT-J。

### 时间数据集（使用 Calendar 工具）

- **TempLAMA**：Calendar 工具使用率仅 0.2%（命名实体太具体，仅知日期不够），主要使用 QA 和 WikiSearch
- **DATESET**：Calendar 工具使用率 54.8%，Toolformer 达 27.3%（GPT-J 仅 3.9%）

### 语言建模能力保持

在 CCNet 和 WikiText-103 上，Toolformer (disabled) 的 perplexity 与 GPT-J + CC 几乎相同（WikiText: 10.3 vs 10.3, CCNet: 10.5 vs 10.5），确认**不损失通用能力**。

### 规模效应

在 GPT-2 系列（124M, 355M, 775M, 1.6B）上的实验表明：
- 工具使用能力在约 **775M 参数**时才开始涌现，更小的模型有无工具表现相当
- Wikipedia Search 是例外——API 使用较简单，更小模型也能受益
- 随模型增大，无工具能力提升的同时，工具使用能力同步提升，两者差距持续存在

### 解码策略分析

不同 $k$ 值对 T-REx 和 WebQS 的影响：
- $k=1$（贪心解码）：模型有一定校准性——选择调用的恰好是它不擅长的问题
- $k=10$：调用率大幅提升（T-REx 从 40.3%→98.1%），但校准性丧失
- $k=3$ 是一个折中选择

## 四、个人评价

Toolformer 是 LLM 工具使用领域的开创性工作，首次实现了 LM 自主学习何时、如何使用工具：

影响：
1. **开创自监督工具学习范式**：不需要人类标注哪些位置该用工具，而是让模型自己判断 API 调用是否有帮助
2. **小模型 + 工具 > 大模型**：6.7B 的 Toolformer 在知识探测和数学推理上超越 175B 的 GPT-3，展示了工具增强的巨大潜力
3. **不损失通用能力**：增强数据集保留了原始文本内容，微调后 LM 本身能力不变
4. **为后续工作奠基**：直接影响了 Gorilla、ToolLLM、HuggingGPT 等 AI Agent 工作
5. **涌现现象**：工具使用能力在约 775M 参数时涌现，与大模型的其他涌现能力一致

局限：
- 工具调用是**一次性**的——模型无法多轮交互（如搜索后调整查询），且每输入最多一次调用
- 仅支持顺序 API 调用，不支持工具间的复杂组合
- BM25 搜索引擎过于简单，限制了搜索增强的效果
- 推理时使用 $k=10$ top-k 解码是人工设定，不够优雅
- 没有处理 API 调用失败或返回无关结果的情况
- 工具数量有限（5 个），扩展到更多工具时的效果未知
- 多语言任务上微调 CCNet 导致分布偏移，性能不如 vanilla GPT-J
- TempLAMA 例子暴露了单次调用限制：理想策略是先查日历再查 QA，但被限制无法实现
