# Mixtral of Experts (Jiang et al., 2024)

**arxiv**: 2401.04088 | **关键词 MoE 开源** | Mistral AI | 2024

---

## 一、核心问题

大语言模型的能力与参数量正相关，但更大的模型意味着更高的推理成本。Mixture-of-Experts（MoE）架构通过稀疏激活实现"大参数量、低推理成本"的折中，但此前的 MoE 模型（如 Switch Transformer、GShard）要么不开源，要么性能未能与稠密大模型竞争。

**核心问题：能否构建一个开源的稀疏 MoE 语言模型，在仅使用 13B 活跃参数的情况下，匹配甚至超越 Llama 2 70B（70B 活跃参数）的性能？**

## 二、方法

### 2.1 架构

Mixtral 8x7B 基于 Mistral 7B 的架构，关键区别是将每层的 FFN 子块替换为 **MoE 层**：

| 参数 | 值 |
|------|-----|
| dim | 4096 |
| n_layers | 32 |
| head_dim | 128 |
| hidden_dim | 14336 |
| n_heads | 32 |
| n_kv_heads | 8 (GQA) |
| context_len | 32768 |
| num_experts | 8 |
| top_k_experts | 2 |

### 2.2 稀疏 MoE 层

对输入 $x$，MoE 层的输出为：

$$y = \sum_{i=0}^{n-1} G(x)_i \cdot E_i(x)$$

路由网络 $G(x)$ 使用 **Top-K Softmax**：

$$G(x) := \text{Softmax}(\text{TopK}(x \cdot W_g))$$

其中 $(\text{TopK}(\ell))_i = \ell_i$ 当 $\ell_i$ 属于 logits 中最大的 $K$ 个坐标，否则为 $-\infty$。

Mixtral 设 $K=2$，每个 token 在每层被路由到 2 个 SwiGLU 专家：

$$y = \sum_{i=0}^{7} \text{Softmax}(\text{Top2}(x \cdot W_g))_i \cdot \text{SwiGLU}_i(x)$$

**参数量分析**：
- 总参数量（稀疏参数）：~47B（8 个专家的 FFN 权重 + 共享的 attention 权重）
- 活跃参数量：~13B（每个 token 只激活 2 个专家 + 全部 attention 权重）
- 注意力权重是**所有专家共享**的，仅 FFN 部分是专家特化的

### 2.3 与 GShard 的区别

- Mixtral 将**每一层**的 FFN 都替换为 MoE 层（GShard 只替换隔层）
- Mixtral 使用简单的 Top-2 路由（GShard 对第二个专家使用更复杂的路由策略）

### 2.4 训练

- 预训练使用多语言数据，上下文长度 32K tokens
- Instruct 版本使用 SFT + DPO（Direct Preference Optimization）微调

**关键发现**：Mixtral 以 5× 更少的活跃参数，在几乎所有基准上**匹配或超越 Llama 2 70B**。尤其在数学和代码生成上大幅领先。

|------|----------|----------|-------------|-------------|
| Llama 2 70B | 64.3% | 64.2% | 66.0% | 65.1% |

Mixtral 在四种欧洲语言上都大幅超越 Llama 2 70B，归因于预训练中多语言数据的上采样。

### 3.4 长上下文性能

- Passkey 检索任务：在 32K 上下文范围内**100% 检索准确率**，不受序列长度和位置影响
- Proof-pile 数据集上困惑度随上下文增长单调下降

### 3.5 Instruct 版本（人类评估）

Mixtral 8x7B – Instruct（SFT + DPO）在 LMSys Chatbot Arena 上：
- Elo 评分 1121，**超过** GPT-3.5-Turbo（1117）、Claude-2.1（1117）、Gemini Pro（1111）、Llama 2 70B chat（1077）
- 是截至 2023 年 12 月**最佳开源权重模型**

### 3.6 路由分析

专家选择的分析揭示：
- **无明显的领域特化**：在 ArXiv、GitHub、PhilPapers、PubMed 等不同领域，专家分配分布非常相似
- **语法级模式**：连续 token 倾向于选择相同专家（时序局部性），尤其在高层次
- DM Mathematics 的分布略有不同，可能因为其合成性质

| 层 | 连续 token 首选专家相同比例 | 随机期望 |
|----|--------------------------|---------|
| Layer 0 | ~14% | 12.5% |
| Layer 15 | ~28% | 12.5% |
| Layer 31 | ~23% | 12.5% |

高层的时序局部性远超随机，对 Expert Parallelism 的负载均衡有影响，但有利于缓存优化。

### 3.7 偏差评估

- BBQ 准确率：Mixtral 56.0% vs Llama 2 70B 51.5%
- BOLD 情感评分：Mixtral 展现更积极、更均衡的情感分布

## 四、个人评价

Mixtral 8x7B 是开源 MoE 大语言模型的里程碑，首次证明了稀疏 MoE 可以在开源场景下与最先进的稠密模型竞争。

影响：
1. **开源 MoE 的标杆**：Apache 2.0 许可证，促进了 MoE 架构在社区中的广泛采用和研究
2. **效率-性能的帕累托最优**：13B 活跃参数匹配 70B 稠密模型，为推理成本敏感的部署提供了新选择
3. **MoE 架构的实证验证**：证明 Top-2 路由 + SwiGLU 专家的简单设计就足够有效，无需复杂的路由策略
4. **路由分析的新洞察**：专家不按领域特化而按语法/位置分配，挑战了"专家 = 领域专家"的直觉
5. **后续影响**：直接催生了 Mixtral 8x22B、DeepSeek-MoE、Qwen-MoE 等一系列开源 MoE 模型

局限：
- **总参数量仍为 47B**：虽然活跃参数少，但推理时仍需将全部参数加载到内存/显存中，对硬件要求不低
- **路由不按领域特化**：这意味着 MoE 的"专家分工"优势可能不如预期，8 个专家可能存在冗余
- **Expert Parallelism 的负载均衡**：时序局部性导致某些专家过载，需要额外的负载均衡机制
- **推理速度**：在单 GPU 上，MoE 层的路由开销和多个专家的内存访问降低了实际加速比
- **后续更优的 MoE 设计**：DeepSeek-V2/V3 的细粒度专家 + 共享专家设计进一步改进了 MoE 架构
