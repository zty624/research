# Relax: Composable Abstractions for End-to-End Dynamic Machine Learning (Lai et al., 2025)

**arxiv**: 2311.02103 | **关键词 ML编译器 动态形状** | CMU / NVIDIA / OpenAI | ASPLOS 2025

## 一、核心问题

现代 ML 模型（特别是 LLM）广泛涉及**动态形状**（dynamic shape）计算：输入序列长度、batch size、KV-cache 长度等在运行时才能确定。现有 ML 编译器在处理动态形状时面临三大挑战：

1. **跨层级优化困难**：计算图（graph level）、张量程序（tensor program level）、外部库三个抽象层级之间是单向、一次性下降（single-shot lowering），无法利用低层级信息指导高层优化
2. **动态形状信息丢失**：现有编译器将动态维度标注为"unknown"，不追踪维度间的符号关系（如一个张量的形状为 $n$，另一个为 $4n$），导致关键优化（如内存规划、算子融合）无法进行
3. **新兴平台部署受限**：移动端、嵌入式设备、WebGPU 等平台需要 AOT（ahead-of-time）编译，但 JIT 方案（如 PyTorch compile）无法支持

**核心问题：如何设计一种跨层级编译器抽象，在全局追踪动态形状的符号关系，从而实现端到端的动态形状感知优化？**

## 二、方法

### 2.1 Relax 抽象概述

Relax 提出两个核心设计：

1. **跨层级抽象（Cross-level Abstraction）**：将计算图、张量程序、外部库调用统一在同一表示中
2. **一阶符号形状（First-class Symbolic Shape）**：用符号变量表示动态维度，全局追踪维度间的符号关系

### 2.2 语言构造

**注解（Annotations）**：Relax 中每个值关联一个结构注解，类似静态类型系统：

| 注解类型 | 示例 | 含义 |
|---------|------|------|
| Shape | `Shape([n, 4])` | 符号形状值 |
| Tensor | `Tensor((n, 4), "f32")` | 带符号形状的张量 |
| Tuple | `Tuple[Tensor((n,4),"f32"), Object]` | 元组 |
| Callable | `Callable([Tensor(("n",4),"f32")], Tensor(("n*4",),"f32"))` | 函数签名 |

**数据流块（Dataflow Block）**：标记无副作用、无控制流的直线程序区域，简化程序变换（如死代码消除）。

**函数调用**：支持同层级调用（图级调用图级）和跨层级调用（图级调用张量程序/外部库）。

### 2.3 一阶符号形状

与 ONNX/Relay 的"unknown"标注不同，Relax 使用符号表达式描述动态维度：

```python
# Unknown 标注——维度信息完全丢失
def any_shape_fn(x: Tensor((?, 2, 2), "f32")):
    n = get_shape_value(x, axis=0)
    lv0: Tensor((?, 4), "f32") = reshape(x, (n, 4))  # ? 无法推断

# 符号形状标注——维度关系被保留
def symbolic_shape_fn(x: Tensor(("n", 2, 2), "f32")):
    n, m = sym_var(), sym_var()
    lv0: Tensor((n, 4), "f32") = reshape(x, shape(n, 4))
    lv1: Tensor((n * 4,), "f32") = flatten(lv0)  # n*4 可以推导
```

符号形状的好处：
- 推导 flatten 后元素总数为 $4n$，与输入相同，暗示**缓冲区复用**
- 内存规划时可以比较符号表达式，判断两个张量是否可以共享存储
- 对于数据依赖算子（如 `unique`），使用 `match_cast` 引入新的符号变量

### 2.4 跨层级抽象

Relax 通过两个跨层级调用原语桥接不同抽象层级：

**`call_tir`**：图级直接调用张量程序（destination-passing style）：

```python
def call_tir(tir_func, args, annotation, sym_args):
    output = alloc_tensor(annotation.shape, annotation.dtype)
    tir_func(*args, output, *sym_args)  # DPS 调用
    return output
```

- 输出形状注解作为参数传入，将图级的符号形状信息**传递**给张量程序
- 张量程序可以特化静态维度，仅在必要时使用动态维度

**`call_dps_library`**：图级直接调用外部库函数（如 cuBLAS、CUTLASS）。

### 2.5 跨层级优化

#### 2.5.1 符号形状推导

- 前向推导：根据算子的注册形状规则，从输入推导输出形状
- 跨函数推导：通过函数签名传播符号关系，无需查看函数体
- 支持参数注解中的符号表达式（如 $2n$），对算子融合等变换至关重要

#### 2.5.2 动态形状感知算子融合

三步流程：
1. **分析反馈**：分析张量程序的循环模式，自动标注 pattern kind（ElementWise / Broadcast / Injective / Reduction / OutputEwiseFusible / Opaque）
2. **FuseOps**：基于 pattern kind 的模式匹配，将算子分组为子图函数，**保留符号形状参数**
3. **FuseTensorIR**：将子图函数中的多个张量程序合并为单一程序

跨层级融合的独特能力：可以融合**自定义张量程序**（如量化解码），这在仅使用图级信息时无法实现。

#### 2.5.3 动态形状感知内存规划

利用符号形状分析，比较动态形状张量的大小关系，实现内存复用：

$$\text{RequestReuseWithSymShape}(shape_1, shape_2) \Rightarrow \text{若 } shape_1 = shape_2 \text{（符号等价），则复用}$$

还可以对符号变量取上界（如用户标注的最大序列长度），实现静态内存分配——这对内存受限设备至关重要。

#### 2.5.4 跨层级张量程序工作空间提升

张量程序中的全局内存分配（如 StreamK matmul 的中间缓冲区）可以**提升到图级**，参与全局内存规划，进一步增加复用机会。

#### 2.5.5 CUDA Graph 卸载

通过静态内存规划预分配所有内存（包括动态形状张量），将 CUDA Graph 从仅支持静态模型扩展到**动态工作负载**。

#### 2.5.6 张量算子优化的部分下降

- 将特定模式（如 matmul + epilogue）部分下降到外部库
- 其余算子使用分析式调度规则或 auto-tuning
- 所有变换可组合，只需一个简单的部分下降 pass 即可定制算子优化

### 2.6 编译流水线

固定顺序管道（无不动点迭代）：

1. 部分库下降（优先利用平台优化库）
2. 生成张量程序 + 降低算子调用为 `call_tir`
3. 算子融合
4. 工作空间提升
5. 内存规划
6. CUDA Graph 卸载
7. 构建可运行模块（虚拟机指令 + GPU 代码）

## 三、实验结果

### LLM 推理性能

在 NVIDIA RTX 4090 上，Llama3-8B 的 decode token 延迟：

- Relax 与 vLLM、llama.cpp 性能相当，最大 **27%** 的延迟降低
- 在 AMD Radeon 7900 XTX 和 Apple M2 Ultra 上也表现竞争力
- Relax 编译一次即可支持任意 batch size 和序列长度

### 优化消融（Llama3-8B, RTX 4090）

| 优化 | 贡献 |
|------|------|
| 部分库下降 | 最大 27%（大 batch 下 cuBLAS 加速 matmul） |
| 算子融合 | 融合约 1/5 算子（RMSNorm + element-wise add） |
| CUDA Graph 卸载 | 1-2%（减少 kernel launch 开销） |

### 内存规划效果

| 设置 | Prefill 激活内存 | Decode 激活内存 |
|------|----------------|----------------|
| 无规划 | 192.7 MiB | 150.0 MiB |
| **有规划** | **149.7 MiB**（↓22%） | **88.2 MiB**（↓40%） |

静态内存规划在连续不同长度/批次的推理中始终复用内存，而运行时内存池在形状变化时反复分配。

### 新兴平台部署

| 设备 | 后端 | Llama3-8B (4-bit) 吞吐 |
|------|------|----------------------|
| iPhone 14 Pro | Metal | 5.1 tok/s |
| Samsung S23 | OpenCL | 7.9 tok/s |
| Orange Pi 5 | OpenCL | 2.3 tok/s |
| Steam Deck | Vulkan | 14.0 tok/s |
| WebGPU (M3 Max) | WebGPU | 37.8 tok/s |

Relax 是首个在这些平台上实现 GPU 加速 LLM 推理的方案（除 Jetson Orin 外）。在 Samsung S23 上比 llama.cpp 快 **55%**（llama.cpp 只用 CPU）。

## 四、个人评价

Relax 是 ML 编译器领域在动态形状处理上的重要突破，将"跨层级优化"从概念推进到了工程实现。

影响：
1. **动态形状的符号追踪**：一阶符号形状的设计使得编译器可以在 AOT 编译时获得足够的形状信息，这对新兴平台（移动端、WebGPU）的部署至关重要
2. **跨层级抽象的工程验证**：证明了将计算图、张量程序、外部库统一表示的实际价值——分析反馈、部分下降、跨层级变换都依赖于这种统一
3. **LLM 的通用部署**：Relax 是少数能在 GPU、AMD GPU、Apple GPU、移动端、WebGPU 上统一部署 LLM 的编译器
4. **已集成到 Apache TVM**：成为主流开源编译器框架的一部分

局限：
- **性能与手写优化仍有差距**：在特定场景下不如专门优化的系统（如 vLLM 在大 batch 推理上的 PagedAttention）
- **符号推导的表达能力有限**：前向推导无法处理复杂的数据依赖形状关系
- **编译时间**：AOT 编译的开销未在论文中详细讨论
- **生态系统**：相比 PyTorch / JAX 等主流框架，Relax 的用户群体和算子覆盖仍然有限
