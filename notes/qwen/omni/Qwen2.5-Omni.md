- **arXiv ID**: 2503.20215
- **开源地址**: https://github.com/QwenLM/Qwen2.5-Omni

## 2. 核心贡献

Qwen2.5-Omni 是一个端到端的统一多模态模型，能够同时感知文本、图像、音频和视频四种模态，并以流式（streaming）方式同时生成文本和自然语音响应。其三大核心贡献如下：

1. **TMRoPE（Time-aligned Multimodal RoPE）位置编码**：提出了一种将时间信息显式编码进旋转位置嵌入的方法，使得音频和视频帧在时间维度上精确对齐，这是多模态协同理解（特别是音视频同步）的关键基础设施。
2. **Thinker-Talker 架构**：将文本生成（Thinker，即大语言模型）和语音生成（Talker，双轨自回归模型）解耦但联合训练，避免了文本和语音输出之间的模态干扰，同时 Talker 直接利用 Thinker 的高维隐表示来实现流式语音生成。
3. **全流式设计**：从编码端（音频/视觉编码器的 block-wise 处理）到解码端（sliding-window DiT 限制感受野），系统性地降低了流式推理的首包延迟，使实时对话成为可能。

## 3. 模型架构详解

### 3.1 整体架构：Thinker-Talker

整个模型采用"大脑-嘴巴"的类比设计：

- **Thinker**（大脑）：本质上是一个 Transformer Decoder，配合音频编码器和视觉编码器，负责处理和理解所有输入模态，生成高维隐表示和文本输出。Thinker 的 LLM 部分基于 Qwen2.5-7B 初始化。
- **Talker**（嘴巴）：一个双轨自回归 Transformer Decoder，灵感来自 Mini-Omni。它从 Thinker 流式接收高维隐表示和已采样的文本 token 嵌入，然后自回归地生成语音 token。关键点在于 Talker 与 Thinker 共享所有历史上下文信息，两者作为一个整体模型进行端到端训练和推理。

这种设计的一个巧妙之处在于：Talker 既需要 Thinker 的高维表示（以隐式获取语气、态度等尚未生成的语义信息），又需要离散的文本 token 嵌入（因为高维表示空间中语义相似但发音不同的词可能具有相近的表示，离散 token 可以消除这种不确定性）。两者缺一不可。

### 3.2 输入感知模块

#### 文本编码
使用 Qwen 的 byte-level BPE tokenizer，词表大小 151,643。

#### 音频编码
- 重采样至 16kHz，提取 128 通道 mel-spectrogram（窗口 25ms，hop 10ms）
- 音频编码器继承自 Qwen2-Audio（初始化自 Whisper-large-v3）
- 每帧音频表示对应约 40ms 原始音频
- 为支持流式推理，将全注意力改为按 2 秒分块的 block-wise 注意力

#### 视觉编码
- 使用 Qwen2.5-VL 的 ViT 编码器，约 675M 参数
- 混合训练图像和视频数据，兼顾图像理解和视频理解
- 动态帧率采样视频以适配音频采样率；每张图像视为两帧相同帧
- 使用 flash attention 和 2x2 token 合并 MLP 层，patch size=14，支持不同分辨率图像打包

### 3.3 TMRoPE 位置编码

这是本文最具技术深度的贡献之一。TMRoPE 将原始的旋转位置嵌入（RoPE）解构为三个分量：**时间（temporal）、高度（height）、宽度（width）**，并为每种模态赋予不同的位置 ID 模式：

| 模态 | temporal ID | height ID | width ID |
|------|-------------|-----------|----------|
| 文本 | 递增（等同 1D-RoPE） | 同 temporal | 同 temporal |
| 音频 | 每 40ms 递增（相同帧内 ID 相同） | 同 temporal | 同 temporal |
| 图像 | 恒定（单张图像时间维度不变） | 按位置递增 | 按位置递增 |
| 视频帧 | 按实际时间动态递增 | 按位置递增 | 按位置递增 |

**关键设计**：当输入是带音频的视频时，视频帧的 temporal ID 根据实际时间动态调整，确保一个 temporal ID 对应 40ms。这保证了音频和视频在时间轴上的精确对齐。

此外，对于带音频的视频输入，还采用了**时间交错（time-interleaving）**方法：每 2 秒为一个 chunk，视觉表示在前、音频表示在后，交错排列。这种排列方式使模型能够同时接收视觉和听觉信息。

### 3.4 生成模块

#### 文本生成
由 Thinker 直接通过自回归采样生成，与标准 LLM 一致，支持重复惩罚和 top-p 采样。

#### 语音生成
- Talker 自回归生成 audio token，不需要与文本做词级别或时间戳级别的对齐——这大大简化了训练数据需求和推理流程。
- 使用自研的 **qwen-tts-tokenizer** 编码语音，可通过因果音频解码器流式解码。

#### 流式 Codec 解码
- 使用 Flow-Matching DiT 模型将 audio code 转换为 mel-spectrogram
- 再通过修改版 BigVGAN 将 mel-spectrogram 重建为波形
- **滑动窗口 block 注意力**：将相邻 code 分组为 block，限制 DiT 感受野为 4 个 block（2 个 lookback + 1 个 lookahead），逐 chunk 生成 mel-spectrum
- 这种设计显著减少了初始包延迟

## 4. 训练数据和方法

### 4.1 预训练（三阶段）

**阶段一：模态对齐（冻结 LLM）**
- LLM 参数来自 Qwen2.5 初始化，冻结不动
- 视觉编码器初始化自 Qwen2.5-VL，音频编码器初始化自 Whisper-large-v3
- 先训练各编码器的 adapter，再训练整个编码器
- 数据：大量音频-文本对和图像-文本对
- 用自然语言 prompt 替代层级标签（继承自 Qwen2-Audio 的做法），提升泛化和指令跟随能力

**阶段二：全参数联合训练**
- 解冻所有参数
- 新增 800B tokens 的图像/视频数据、300B tokens 的音频数据、100B tokens 的带音频视频数据
- 引入更大规模的多模态混合数据和更丰富的任务类型
- 纯文本数据也参与训练以维持语言能力
- 最大序列长度限制为 8192 tokens

**阶段三：长序列扩展**
- 引入长音频和长视频数据
- 将文本、音频、图像、视频数据扩展到 32,768 tokens 长度
- 实验表明对长序列理解有显著提升

### 4.2 后训练（Post-training）

#### Thinker 后训练
- 使用 ChatML 格式的指令微调数据
- 数据涵盖纯文本对话、视觉模态对话、音频模态对话和混合模态对话

#### Talker 三阶段训练

1. **ICL（In-Context Learning）阶段**：学习上下文续接。除文本监督外，通过 next-token prediction 执行语音续接任务，学习从语义表示到语音的单调映射，并习得上下文合适的韵律、情感、口音等多样属性。还使用音色解缠技术防止模型将特定声音与稀有文本模式关联。

2. **DPO 阶段**：构建三元组数据 $(x, y_w, y_l)$，基于 WER 和标点停顿错误率作为奖励分数排序正负样本，通过 DPO 优化提升语音生成稳定性，减少幻觉。

3. **说话人微调阶段**：在上述基础模型上进行特定说话人微调，使 Talker 能够采用特定音色并提升自然度。

## 5. 主要实验结果

### 5.1 文本理解（Text→Text）

| 基准 | Gemma2-9B | Llama3.1-8B | Qwen2-7B | Qwen2.5-7B | Qwen2.5-Omni-7B |
|------|-----------|-------------|----------|------------|-----------------|
| MMLU-Pro | 52.1 | 48.3 | 44.1 | **56.3** | 47.0 |
| MATH | 34.3 | 36.4 | 30.8 | **79.9** | 52.9 |
| GSM8K | 85.7 | 91.6 | 88.7 | **84.8** | 70.4 |

文本性能介于 Qwen2-7B 和 Qwen2.5-7B 之间，这说明多模态能力的引入对纯文本能力有一定牺牲，但仍在可接受范围内。不过 MATH 和 GSM8K 上的差距较大（相比 Qwen2.5-7B 下降约 14-15 个点），这可能是因为多模态训练数据"稀释"了数学推理能力。

### 5.2 音频理解（Audio→Text）

- **ASR**：在 Librispeech test-clean 上 WER 1.8%，接近 Whisper-large-v3 的 1.8% 和 MinMo 的 1.7%；在 CommonVoice 上全面超越 Qwen2-Audio
- **S2TT**：CoVoST2 上 en-de 25.1、de-en 33.9、en-zh 41.5、zh-en 15.7，全面超越 Qwen2-Audio 和 Qwen-Audio
- **音频推理 MMAU**：Sound 67.87、Music 69.16、Speech 59.76、Avg **65.60**，大幅超越 Gemini-Pro-V1.5 的 54.90 和 Qwen2-Audio 的 49.20
- **VoiceBench**：平均分 **74.12**，显著超越 MiniCPM-o 的 71.69 和 Baichuan-Omni-1.5 的 71.14

### 5.3 语音指令跟随（关键亮点）

| 基准 | Qwen2-7B（文本） | Qwen2-Audio（语音） | Qwen2.5-Omni-7B（语音） |
|------|-------------------|---------------------|------------------------|
| MMLU* | 69.3 | 33.2 | 65.6 |
| GSM8K* | 82.3 | 18.4 | 85.4 |
| Math23K* | 92.3 | 23.0 | 87.1 |

这是论文最令人印象深刻的结果之一：Qwen2.5-Omni 用语音指令的 MMLU 达到 65.6，远超 Qwen2-Audio 的 33.2；GSM8K 更是达到 85.4，甚至略超文本输入的 Qwen2-7B 的 82.3。这说明 Qwen2.5-Omni 的端到端语音理解能力已经接近文本理解水平，是语音交互领域的一个重要突破。

### 5.4 图像理解（Image→Text）

- MMMU 59.2 vs Qwen2.5-VL-7B 的 60.0（接近）
- MathVista 67.9 vs Qwen2.5-VL-7B 的 68.2（接近）
- MMBench-V1.1-EN 81.8 vs 82.6（接近）
- DocVQA 95.2 vs 95.7（接近）
- OCRBench_V2 57.8 vs 56.3（甚至更好）

整体与 Qwen2.5-VL-7B 基本持平，远超其他开源 Omni 模型，在多数基准上超过 GPT-4o-mini。

### 5.5 视频理解（Video→Text）

| 基准 | GPT-4o-mini | Qwen2.5-VL-7B | Qwen2.5-Omni-7B |
|------|-------------|---------------|-----------------|
| Video-MME w/o sub | 64.8 | 65.1 | **65.1** |
| Video-MME w/ sub | - | 71.6 | **71.6** |
| MVBench | 64.3 | 72.4 | **70.3** |
| EgoSchema | - | 68.6 | **69.6** |

视频理解能力与 Qwen2.5-VL-7B 基本持平或略有差距，但全面超越其他开源 Omni 模型和 GPT-4o-mini。

### 5.6 多模态协同理解（Multimodality→Text）

**OmniBench** 是论文最重要的综合多模态基准：

| 模型 | Speech | Sound Event | Music | Avg |
|------|--------|-------------|-------|-----|
| Gemini-1.5-Pro | 42.67% | 42.26% | 46.23% | 42.91% |
| MiniCPM-o | - | - | - | 40.5% |
| Baichuan-Omni-1.5 | - | - | - | 42.9% |
| **Qwen2.5-Omni-7B** | **55.25%** | **60.00%** | **52.83%** | **56.13%** |

平均 56.13%，大幅领先第二名（Gemini-1.5-Pro 的 42.91%），这是本文在多模态协同理解方面的核心成果。

### 5.7 语音生成（X→Speech）

**零样本语音生成（SEED 基准）**：

| 模型 | WER test-zh | WER test-en | WER test-hard |
|------|-------------|-------------|---------------|
| Seed-TTS ICL | 1.11 | 2.24 | 7.58 |
| Seed-TTS RL | **1.00** | **1.94** | 6.42 |
| MaskGCT | 2.27 | 2.62 | 10.27 |
| CosyVoice 2 | 1.45 | 2.57 | 6.83 |
| Qwen2.5-Omni ICL | 1.70 | 2.72 | 7.97 |
| **Qwen2.5-Omni RL** | 1.42 | 2.33 | **6.54** |

RL 优化后的版本在 test-hard 上达到 6.54% WER，超越 CosyVoice 2 的 6.83% 和 MaskGCT 的 10.27%，仅略逊于 Seed-TTS RL。考虑到这是一个统一模型中的语音生成模块，而非专门的 TTS 模型，这个结果非常出色。

**单说话人生成（说话人微调后）**：NMOS 评分达到 4.48-4.62（中文）/4.58-4.62（英文），接近人类水平的 4.51/4.46。

## 6. 与前代/竞品的对比

### 与 Qwen 家族前代对比

| 维度 | Qwen2-Audio | Qwen2.5-VL | Qwen2.5-Omni |
|------|-------------|------------|--------------|
| 支持模态 | 文本+音频 | 文本+图像+视频 | 文本+图像+音频+视频 |
| 语音输出 | 无 | 无 | 有（流式） |
| 音频理解 | 基准水平 | N/A | 全面超越 Qwen2-Audio |
| 视觉理解 | N/A | 基准水平 | 基本持平 Qwen2.5-VL |
| 文本能力 | - | 基准水平 | 介于 Qwen2-7B 和 Qwen2.5-7B 之间 |

Qwen2.5-Omni 的定位很明确：它不是要替代 Qwen2.5-VL 或 Qwen2-Audio 这样的单模态专精模型，而是在一个统一框架下尽可能缩小与它们的差距，同时提供它们无法提供的多模态协同理解和语音生成能力。

### 与同期竞品对比

- **vs MiniCPM-o**：在 VoiceBench（74.12 vs 71.69）、MMAU（65.60 vs 估计约 50+）、OmniBench（56.13% vs 40.5%）上全面领先；但在部分视觉基准上 MiniCPM-o 偶尔占优
- **vs Baichuan-Omni-1.5**：VoiceBench（74.12 vs 71.14）和 OmniBench（56.13% vs 42.9%）上大幅领先
- **vs GPT-4o-mini**：图像和视频理解上基本持平或超越
- **vs Seed-TTS**：语音生成质量接近，但 Qwen2.5-Omni 是统一模型而非专门 TTS

## 7. 局限性和未来方向

### 论文承认的局限性

1. **文本能力退化**：多模态训练导致纯文本能力（尤其是数学推理）有所下降，MATH 从 79.9 降到 52.9，GSM8K 从 84.8 降到 70.4
2. **视频 OCR 和音视频协同理解**：论文在结论中承认这些是被忽视的重要问题，需要学术界和工业界合作建设更全面的评估基准和研究数据集
3. **语音说话人相似度（SIM）**：在 SEED test-en 上 SIM 为 0.641，低于 Seed-TTS RL 的 0.766，说明零样本语音克隆的说话人保真度仍有提升空间
4. **模型规模**：仅发布了 7B 版本，更大规模的版本尚未公开

### 未来方向（论文提及）

1. 开发更鲁棒、更快速的模型
2. 扩展输出模态到图像、视频、音乐等
3. 构建更全面的多模态评估基准

### 笔者观察到的潜在问题

1. **训练数据细节不足**：论文只给出了 token 数量，未详细说明数据来源、质量过滤策略和具体配比，这使得复现难度很大
2. **流式延迟数据缺失**：论文提出了多种降低首包延迟的设计（block-wise 编码、sliding-window DiT），但没有报告实际的延迟数字（如 TTFT、首包延迟等），无法定量评估这些设计的实际效果
3. **Talker 与 Thinker 的联合训练机制**：论文描述了三阶段 Talker 训练，但没有详细说明联合训练中是否会出现梯度冲突、训练不稳定等问题，也没有消融实验验证 Thinker-Talker 架构相比简单拼接方案的优势
4. **TMRoPE 的消融研究**：TMRoPE 是核心贡献之一，但论文没有提供消融实验来证明它比简单的位置拼接或扁平化方案好多少

## 8. 个人见解和启发

### 8.1 架构设计的哲学思考

Thinker-Talker 架构的设计哲学——"分而治之但联合优化"——很值得借鉴。它避免了在单一输出空间中同时建模文本和语音 token 的困难，而是将两者分配给专门的模块，但通过共享高维表示保持信息流通。这与人类大脑的分工有异曲同工之妙：布洛卡区负责语言产生，韦尼克区负责语言理解，但两者通过神经纤维束紧密相连。

### 8.2 TMRoPE 的意义

TMRoPE 是一个简洁而有效的设计。它不是简单地拼接位置 ID，而是通过解构 RoPE 为三维分量，让不同模态可以在时间维度上自然对齐。这种"正交分解"的思路可能适用于更多需要跨模态时间同步的场景，比如音乐与舞蹈的联合生成、多摄像头视频的时间对齐等。

### 8.3 语音指令跟随的突破

Qwen2.5-Omni 在语音指令跟随上的表现（GSM8K 语音 85.4 vs 文本 82.3）令人意外。这说明端到端的语音理解（而非 ASR+LLM 的级联方式）可能具有独特优势：语音中的韵律、停顿、重音等信息可能为推理提供了额外线索。这值得进一步研究——端到端模型是否真的在"听"而不仅仅是"转录"？

### 8.4 对视频生成领域的启发

对于我们做视频生成的工作，Qwen2.5-Omni 提供了几点启发：

1. **时间对齐是关键**：TMRoPE 的思路可以迁移到视频生成中，让模型更精确地控制音视频的时间同步
2. **Block-wise 处理是长序列的出路**：视频生成长度通常受限，block-wise 编码+流式解码的范式可能适用于长视频生成
3. **统一模型 vs 专用模型**：虽然统一模型在各单项上不如专用模型，但多模态协同理解带来的涌现能力（如 OmniBench 上的大幅领先）说明统一建模有其独特价值
4. **流式输出的工程价值**：在交互式应用场景中，首包延迟比总延迟更重要，sliding-window 的设计思路可以直接复用

### 8.5 总体评价

Qwen2.5-Omni 是当前开源 Omni 模型中最为全面和强大的。它在多模态协同理解上大幅领先竞品，语音指令跟随接近文本水平，语音生成质量接近专用 TTS 模型。其主要代价是纯文本能力（尤其数学推理）的下降，以及论文在技术细节（数据、延迟、消融）上的不足。总体而言，它代表了向真正的多模态统一智能迈出的坚实一步。
