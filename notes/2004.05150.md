# Longformer: The Long-Document Transformer (Beltagy et al., 2020)

**arxiv**: 2004.05150 | **长文档 Transformer** | Allen AI | 2020

---

## 一、核心动机

标准 Transformer 的自注意力机制复杂度为 $O(N^2)$，其中 $N$ 为序列长度。这导致 BERT 等模型只能处理 512 个 token 的序列，而现实中的许多任务涉及远超此长度的文档：

- 问答（文档可能数千词）、文档分类、摘要生成等任务的输入往往超过 4096 token
- 强行截断会丢失长距离上下文，严重损害模型性能
- 使用滑动窗口分片处理则无法跨片段建模依赖关系

核心问题：**如何让 Transformer 高效处理长文档，同时保留捕获长距离依赖的能力？**

已有方案如 Sparse Transformer 和 Transformer-XL 各有局限：Sparse Transformer 的稀疏模式不够灵活，Transformer-XL 依赖缓存机制而非真正的长距离注意力。

## 二、关键创新

### 2.1 滑动窗口注意力 (Sliding Window Attention)

每个 token 只关注其周围 $w$ 个邻居：

$$\text{Attention}_{local}(i) = \text{softmax}\left(\frac{Q_i K_{[i-w/2 : i+w/2]}^T}{\sqrt{d_k}}\right) V_{[i-w/2 : i+w/2]}$$

- 复杂度：$O(N \times w)$，当 $w \ll N$ 时远优于 $O(N^2)$
- 多层堆叠后，感受野随层数线性增长：$L$ 层的有效感受野为 $L \times w$
- 灵感来源：CNN 的局部感受野 + 多层扩展；语言学中局部上下文通常最重要

### 2.2 全局注意力 (Global Attention)

仅对少量关键 token（如 [CLS]、问题 token）使用全局注意力：

- **全局 token**：attend 到序列中所有 token，且被所有 token attend 到
- **普通 token**：只使用滑动窗口注意力
- 全局注意力的计算：为全局位置使用独立的 $Q_g, K_g, V_g$ 投影（与局部注意力的 $Q, K, V$ 分离）

这种设计的核心洞察：**大部分信息可以通过局部窗口捕获，只有少数 token 需要全局视野来建立长距离连接**。

### 2.3 稀疏注意力模式

Longformer 的注意力模式可以看作三种模式的组合：

1. **Sliding window**：捕获局部上下文（$w=512$）
2. **Dilated sliding window**（可选）：类似膨胀卷积，在不增加计算量的前提下扩大感受野
3. **Global attention**：由任务决定的少量全局 token，捕获长距离依赖

整体注意力矩阵是高度稀疏的，但信息可以通过全局 token 和多层堆叠的局部窗口在整个序列中传播。

### 2.4 高效实现

- 滑动窗口注意力使用**自定义 CUDA kernel** 实现，避免实例化 $N \times N$ 的注意力矩阵
- 全局注意力使用标准矩阵乘法（因为全局 token 数量少，矩阵很小）
- 支持 attention mask，可以灵活地为不同任务配置不同的全局 token

## 三、方法细节

### 与 BERT 的关系

Longformer 从 BERT 的预训练权重初始化，然后继续在长文档上预训练，实现从短到长的迁移。这比从头训练高效得多。

### 全局 token 的选择

全局 token 的选择取决于下游任务：

- **分类**：[CLS] token 设为全局
- **问答**：问题部分的所有 token 设为全局，文档部分使用滑动窗口
- **序列标注**：不需要全局 token，纯滑动窗口即可

### 模型配置

| 模型 | 层数 | $d_{model}$ | 注意力头数 | 参数量 |
|------|------|------------|-----------|--------|
| Longformer-base | 12 | 768 | 12 | 149M |
| Longformer-large | 24 | 1024 | 16 | 433M |

参数量略大于 BERT，因为额外的全局注意力投影 $Q_g, K_g, V_g$。

### 预训练策略

- 从 BERT checkpoint 初始化
- 在长文档语料上继续 MLM 预训练，序列长度 4096
- 滑动窗口大小 $w = 512$

## 四、实验结果

### 长文档分类 (6 个数据集)

| 数据集 | 文档平均长度 | RoBERTa | Longformer |
|--------|------------|---------|------------|
| IMDb | 876 | 95.3 | 95.6 |
| Hyperpartisan | 742 | 78.2 | **91.8** |
| 20Newsgroups | 370 | 84.2 | 85.3 |
| ECtHR | 2,243 | 80.4 | **87.0** |
| CMU-AQP | 4,834 | 63.6 | **73.5** |
| Multi-XScience | 5,947 | 71.7 | **78.1** |

在文档越长、长距离依赖越重要的数据集上，Longformer 的优势越大。Hyperpartisan 上提升超过 13 个百分点，说明标准模型截断长文档会严重丢失关键信息。

### 问答 (WikiHop, TriviaQA)

- **WikiHop**（多文档推理）：Longformer-large 达到 74.3，显著优于 RoBERTa 的 68.8
- **TriviaQA**（长上下文阅读理解）：在长文档设置下明显优于基线

### 与其他长序列模型的比较

| 模型 | 复杂度 | 最大序列长度 | WikiHop |
|------|--------|------------|---------|
| RoBERTa | $O(N^2)$ | 512 | 68.8 |
| Transformer-XL | $O(N^2)$ | 512+缓存 | 67.6 |
| Sparse Transformer | $O(N\sqrt{N})$ | 4096 | — |
| **Longformer** | $O(N \times w)$ | **4096+** | **74.3** |

Longformer 在效率和性能之间取得了更好的平衡。

### 关键实验发现

1. **全局注意力至关重要**：移除全局注意力后，问答任务性能显著下降
2. **滑动窗口大小的影响**：$w=512$ 是较好的平衡点，过小损失局部信息，过大增加计算量
3. **从 BERT 初始化很重要**：从头训练的效果不如从预训练权重迁移

## 五、影响

### 直接影响

Longformer 证明了**局部注意力 + 少量全局注意力**的组合可以高效处理长文档，成为长序列 Transformer 的标准范式之一。

### 重要后续工作

1. **BigBird** (Zaheer et al., 2020)：类似思路，增加了随机注意力模式，提供了更好的理论保证（图灵完备性证明）
2. **LongT5** (Guo et al., 2022)：将长文档注意力思想融入 T5 架构
3. **LED (Longformer Encoder-Decoder)**：Longformer 的 encoder-decoder 版本，用于长文档摘要
4. **FlashAttention** (Dao et al., 2022)：从硬件效率角度解决注意力瓶颈，最终使全注意力处理长序列成为可能
5. **Ring Attention**：分布式长序列注意力，支持百万级 token

### 更广泛的意义

- **效率与表达力的权衡**：Longformer 的核心思想——大部分信息是局部的，少量全局连接就够——是一个具有普遍意义的设计原则
- **任务自适应的注意力模式**：不同任务需要不同的全局 token 选择，这种灵活性是 Longformer 相对固定稀疏模式的优势
- **从短到长的迁移**：从 BERT 初始化再继续预训练的策略，成为后续长上下文模型的标准做法

### 局限性

1. **自定义 CUDA kernel**：实现复杂，可移植性差，阻碍了广泛采用；后续 FlashAttention 的 io-aware 实现方式更优雅
2. **全局 token 需要手工选择**：不同任务需要不同的全局 token 配置，缺乏自动选择机制
3. **4096 仍然不够长**：对于超长文档（书籍级别，10 万+ token），Longformer 仍然不够。FlashAttention + KV cache 压缩等技术最终提供了更好的解决方案
4. **最终被 FlashAttention 取代**：FlashAttention 通过硬件感知的精确注意力实现，使得在 8K-128K 序列长度上使用全注意力成为可能，Longformer 的稀疏近似不再是必要妥协
